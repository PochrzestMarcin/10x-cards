<database_planning_summary>
The MVP database will consist of three core tables:
cards – stores accepted flashcards with UUID PK, user_id FK (auth.users), front VARCHAR(200), back VARCHAR(500), created_at, updated_at; indexed by (user_id, created_at DESC).
generation_requests – captures each AI generation attempt with UUID PK, user_id FK, source_text TEXT (1 000–10 000 chars), optional model parameters (model, temperature, top_p), status ENUM, optional duration_ms, created_at.
generation_errors – logs failures with UUID PK, user_id FK, nullable request_id FK, error_code ENUM, error_message, metadata JSONB, created_at.
Relationships:
auth.users.id → cards.user_id, generation_requests.user_id, generation_errors.user_id (all ON DELETE CASCADE).
generation_requests.id → generation_errors.request_id (nullable, ON DELETE CASCADE).
Security:
Per-table RLS enforcing user_id = auth.uid().
Service-role bypass for background/admin operations.
Constraints & Data Types:
UUID PKs, VARCHAR length limits, ENUMs for status and error_code.
Timestamps stored as timestamptz (UTC).
Hard deletes only; no soft-delete columns.
Scalability & Performance:
Expect small data volume; no partitioning.
Keyset pagination enabled by (user_id, created_at) index.
No full-text search, scheduling, or tagging in MVP.
Analytics:
Acceptance metrics and error counts derived at query time; no precomputed aggregates.
This schema satisfies the PRD while remaining minimal, secure, and extensible for future features such as decks, scheduling, sharing, and analytics.
</database_planning_summary>
